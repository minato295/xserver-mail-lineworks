# Xserver受信メール LINE WORKS通知 設計

## 目的

Xserver上の設定対象メールアカウントが受信したメールを即時に解析し、会社のLINE WORKSトークルームへ読みやすい通知として送る。メール原本は通常のメールボックスにも残す。対象メールアドレスはコードや公開リポジトリへ保存せず、導入時にXserver側で指定する。

## 構成

Xserverの「メールの振り分け」で対象メールを公開領域外のPHP CLIへ渡す。振り分けアクションはコピー転送とし、PHP処理やLINE WORKS送信が失敗してもメール原本を失わない。

PHP、依存パッケージ、秘密設定、実行ログはすべて `public_html` の外へ配置する。公開領域にはPHP、Webhook URL、ログ、メール本文、一時ファイルを置かない。

## データフロー

1. 設定対象のメールアカウントがメールを受信する。
2. Xserverのメール振り分けがメール原文をPHP CLIの標準入力へ渡す。
3. PHPが入力サイズを制限しながらメール原文を読み取る。
4. MIMEヘッダー、文字コード、multipart構造を解析する。
5. `text/plain` を優先し、存在しない場合は `text/html` を安全なプレーンテキストへ変換する。
6. LINE WORKS Incoming Webhook向けJSONを生成し、HTTPS POSTする。
7. 成功・失敗の最小限の運用情報だけを公開領域外のログへ記録する。

添付ファイル、インライン画像、実行可能コンテンツはLINE WORKSへ送らない。

## LINE WORKSの表示形式

通常は、メール1通につきLINE WORKSのメッセージ1件で送信する。項目順は必ず次の通りとする。

```text
受信日時：2026年07月11日（土）16時05分09秒
From：sender@example.invalid
To：notification@example.invalid
Cc：manager@example.invalid
Bcc：hidden@example.invalid
件名：お問い合わせについて
本文：
メール本文……
```

- 受信日時はPHP処理開始時刻ではなく、受信メールの `Date` ヘッダーを解析して日本時間 `Asia/Tokyo` に変換する。
- 日時形式は `yyyy年MM月dd日（EEE）HH時mm分ss秒` とし、曜日は `日・月・火・水・木・金・土` の日本語1文字を用いる。
- `Date` ヘッダーが欠損または不正な場合は、Xserver上でPHPがメールを受け取った時刻を使う。
- From、To、Cc、Bcc、件名はMIMEエンコードをデコードする。
- To、Cc、Bccが複数ある場合はすべて表示する。
- Ccが空なら `Cc` 行を表示しない。
- Bccが空または配送時に削除されて取得できなければ `Bcc` 行を表示しない。Bccを推測しない。
- 本文はテキストとして全文を送る。HTMLタグ、スクリプト、スタイルは除去する。
- メール末尾の署名や引用文も省略しない。

Incoming Webhookがメッセージサイズ超過として単一送信を拒否した場合だけ、本文を欠落させないため複数メッセージへ分割する。分割時は各メッセージに `(1/3)` のような連番を付ける。通常は分割しない。

## Webhook通信

- LINE WORKS Incoming Webhookの公式JSON形式 `title` と `body.text` を使用する。
- Webhook URLはコードへ直書きせず、公開領域外の権限 `600` の設定ファイルから読む。
- HTTPS証明書を検証する。検証無効化オプションを使用しない。
- 接続タイムアウトと総タイムアウトを設定する。
- HTTP 2xxだけを成功として扱う。
- タイムアウト、HTTPエラー、JSON生成失敗を安全に処理する。
- LINE WORKSのレート制限120リクエスト/分を考慮する。
- Webhook URL、メール本文、認証情報をログへ出力しない。

## エラー処理

メール受信処理を阻害しないよう、PHPは内部エラーをログへ記録した後、メール振り分けからの実行に対して安全に終了する。

障害通知は次の優先順位とする。

1. MIME解析、入力検証、JSON生成などの内部エラーでは、Webhook通信が利用可能ならLINE WORKSへ本文・秘密値を含まない障害通知を送る。
2. LINE WORKS Incoming Webhookへの接続失敗、タイムアウト、HTTPエラー、サイズ超過フォールバックの最終失敗の場合だけ、登録済みのエラー通知先へメールを送る。
3. エラーメール送信にはXserver公式の `/usr/sbin/sendmail` を直接使用する。外部SMTPサーバー、SMTPユーザー名、SMTPパスワードは使用しない。
4. LINE WORKS通知が成功した場合はエラーメールを送らない。
5. 同一分類のエラー通知は一定時間抑制し、通知の大量発生を防ぐ。

エラーメールには通知対象メールの本文、Webhook URL、APIキー、サーバー認証情報を含めない。件名、発生日時、エラー分類、Message-IDの一方向ハッシュ、確認すべきログの場所だけを含める。エラー通知先自身がLINE WORKS通知対象に登録されていても再帰しないよう、専用ヘッダーを付与し、メール振り分け条件から除外する。

ログには次だけを記録する。

- 発生日時
- 成否
- Message-IDの一方向ハッシュ（Message-ID原文は記録しない）
- HTTPステータスまたは分類済みエラーコード

再送キューにメール本文を永続保存しない。原本はコピー転送されたメールボックスを正本とする。

LINE WORKSとローカルsendmailの両方が失敗した場合はログだけが残る。この状態を早期発見できるよう、Mac管理CLIにLINE WORKSテスト、エラーメールテスト、ログ診断を用意する。

## Mac管理CLI

通知対象メールアドレスとエラー通知先メールアドレスはコードへ埋め込まず、Mac上の日本語対話式CLIから管理する。Codexを使わず、利用者自身で次を実行できることを要件とする。

- LINE WORKS通知対象の一覧・追加・変更・削除・複数登録
- エラー通知先の一覧・追加・変更・削除・複数登録
- Xserver上の設定との同期状態確認
- LINE WORKSテスト通知
- エラーメールのテスト送信
- 最新ログの秘密値を含まない診断表示

通知対象の変更はXServer APIのメール振り分け設定へ反映する。変更前に現在のルールと差分を表示し、利用者の確認後に追加・変更・削除する。既存の無関係な振り分けルールは変更しない。

エラー通知先の変更は、FTPSを使って公開領域外の権限 `600` の設定ファイルへ反映する。設定ファイルを更新する際は一時ファイルへ安全に書き、権限を設定した後に置換する。実メールアドレスをローカルGit、GitHub、コマンド履歴、ログへ保存しない。

Mac管理CLIはmacOSキーチェーンに登録済みのXServer APIキーとFTPS認証情報を利用する。iPhone/Safari用の管理画面は今回の対象外とする。

## セキュリティ

- 標準入力のメール内容をシェルコマンド、PHPコード、ファイルパスとして評価しない。
- シェルを介さずPHP CLIを直接実行する。
- 最大入力サイズを設定し、過大メールによるメモリ枯渇を防ぐ。
- MIME解析は添付データをLINE WORKS本文へ混入させない。
- 制御文字を除去し、LINE WORKS表示を壊す文字列を正規化する。
- Webhook URLをGit、テストfixture、ログ、エラーメッセージへ含めない。
- エラー通知先メールアドレスをGit、公開fixture、ログへ含めない。
- 設定ファイル・ログディレクトリ・PHP実行ファイルの権限を最小化する。
- テスト用Webhook URLは秘密値として扱い、本番切替時は設定ファイルだけを交換する。

PHPは本セッションのセキュリティスキルが直接サポートする言語に含まれないため、XserverとLINE WORKSの公式仕様、および一般的なPHP/MIME/秘密管理の安全策を基準にする。

## Xserver設定

- 対象: 導入時にXserver側で指定するメールアカウント（公開リポジトリには保存しない）
- 条件: 対象メールアカウントへの受信
- アクション: コマンドへのコピー転送
- コマンド: Xserverのコマンドパス一覧で確認したPHP CLIと、公開領域外の通知PHPの絶対パス
- 原本: 通常のメールボックスへ残す

設定前に現在のメール振り分けルールを取得し、重複や優先順位の衝突がないことを確認する。既存ルールを無断で削除・変更しない。

## テストと導入

1. ローカルで、プレーンテキスト、日本語MIME、HTML、multipart、Cc/Bcc有無、長文、壊れたDate、過大入力を自動テストする。
2. Webhook送信を偽HTTPクライアントへ差し替え、秘密値非露出とエラー処理をテストする。
3. 公開領域外へ配置し、権限とPHP CLIの実行可否を確認する。
4. 合成メールを標準入力へ渡し、テスト用Webhookへ1件通知する。
5. Xserver APIで既存ルールを再確認し、コピー転送ルールを追加する。
6. テスト用WebhookでLINE WORKS通知成功時にエラーメールが送られないことを確認する。
7. Webhook失敗を模擬し、設定済みエラー通知先へsendmailで通知されることを確認する。
8. 外部から設定対象アドレスへテストメールを1通送信し、メールボックスとLINE WORKSの両方で受信を確認する。
9. `public_html` 以下にPHP、設定、ログ、一時ファイルが存在しないことを再確認する。

## 成功条件

- メール原本が設定対象アカウントのメールボックスへ残る。
- LINE WORKS通知が原則1メッセージで即時に届く。
- 指定順序と日本語日時形式でヘッダー情報が表示される。
- Cc/Bccの空行を表示しない。
- テキスト本文全文を表示し、添付ファイルを送らない。
- PHP、Webhook設定、ログが `public_html` 外にあり、外部HTTPアクセスできない。
- Webhook URLとメール本文がログやリポジトリへ残らない。
- LINE WORKSへ障害通知できない場合だけ、設定済みエラー通知先へメールが届く。
- Mac管理CLIから通知対象とエラー通知先を複数管理できる。
